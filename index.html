Cliente de Red y AWS

Retrofit · OkHttp · Lambda

Archivos Clave

  • RetrofitClient.kt – Configuración HTTP
  • ApiService.kt – Contrato de endpoints
  • AWSRepositorio.kt – Encapsula llamadas
  • TiendaRepositorioImpl.kt – Sincronización
  • Lambda (Node.js) – Lógica en DynamoDB

Conceptos Técnicos

  • loggingInterceptor – Depuración en Logcat
  • OkHttpClient – connectTimeout readTimeout
  • GsonConverterFactory – Traduce JSON ↔ Kotlin
  • @POST / @Path – Peticiones HTTP
  • suspend fun – Asíncrono, no bloquea UI
  • Result<T> – Contenedor de éxito/error

Detalle Clave

ID Idéntico: El ID generado por Room se envía a AWS como primary key en DynamoDB.
Lambda: PutCommand / UpdateCommand para insertar y actualizar stock.
try-catch: Captura errores de red y AWS.
Guion (2 min)
Explico la capa de red y la integración con AWS. Configuramos Retrofit con timeouts y Gson para traducir JSON a objetos Kotlin. El repositorio AWSRepositorio envuelve las llamadas en un contenedor Result para manejar fallos. En TiendaRepositorioImpl sincronizamos Room con la nube usando el mismo ID generado localmente, lo que garantiza que al borrar o editar un registro sepamos exactamente qué elemento modificar en DynamoDB. En la Lambda, usamos PutCommand y UpdateCommand para gestionar productos y pedidos.

Interfaz de Usuario y Experiencia (UI/UX)

Jetpack Compose · Material Design 3

Archivos Clave

  • pantalla/InicioPantalla.kt – Pantalla principal
  • pantalla/VentaPantalla.kt – Flujo de ventas
  • MainActivity.kt – Contenedor y tema
  • ui/theme/ – Colores, tipografía, Material 3

Conceptos Técnicos

  • Composables – UI declarativa, recomposición automática
  • Scaffold – Estructura con TopBar y FAB
  • LazyColumn – Lista eficiente (similar a RecyclerView)
  • Material Design 3 Jetpack Compose
Guion (2 min)
Presento la Capa de Presentación Visual. Usamos Jetpack Compose, el kit moderno de Android. A diferencia de XML, la UI declarativa reduce errores y facilita interfaces dinámicas. En VentaPantalla.kt aplicamos componentes de Material Design 3 para que la app sea intuitiva. Destaco el uso de LazyColumn, que permite desplazar cientos de productos sin perder rendimiento porque solo renderiza los elementos visibles. Mi objetivo fue una interfaz limpia y ágil para el administrador.

Lógica de Estado y Navegación

MVVM · StateFlow · NavGraph

Archivos Clave

  • viewmodel/InventarioViewModel.kt – Lógica de inventario
  • viewmodel/VentaViewModel.kt – Lógica de ventas
  • navegacion/NavGraph.kt – Rutas y flujo
  • state/ProductoUiState.kt – Estados de la UI

Conceptos Técnicos

  • StateFlow / MutableStateFlow – Notifica cambios a la UI
  • viewModelScope – Procesos seguros, se cancelan al destruir
  • NavHost – Decide pantalla según ruta
  • MVVM ViewModel
Guion (2 min)
Mi parte cubre la Lógica de Estado y Navegación. Aplicamos MVVM: los ViewModels conectan los datos con la interfaz. Usamos StateFlow para observar cambios en tiempo real; si se agrega un producto, la pantalla se actualiza sin refrescar. Para la navegación, implementamos un NavGraph centralizado que maneja de forma segura el paso de parámetros y asegura que el usuario siga un flujo lógico, como no poder entrar al inventario sin pasar por el Login.
4

Reglas de Negocio (Capa de Dominio)

Use Cases · Clean Architecture

Archivos Clave

  • domain/modelo/Producto.kt – Entidades puras
  • domain/usecase/RegistrarVentaUseCase.kt – Acción de negocio
  • domain/usecase/ConfirmarPedidoUseCase.kt – Lógica de pedidos
  • domain/repositorio/TiendaRepositorio.kt – Interfaz de reglas

Conceptos Técnicos

  • Casos de Uso – Cada archivo = una acción de negocio
  • Modelos de Dominio – Sin dependencias externas
  • Abstracción – El dominio no sabe de dónde vienen los datos
  • Clean Architecture Independencia
Guion (2 min)
Presento la Capa de Dominio, el corazón de la lógica de negocio. Siguiendo Clean Architecture, esta capa es totalmente independiente de la base de datos o de internet. Aquí definimos los Casos de Uso. Por ejemplo, RegistrarVentaUseCase contiene la lógica exacta de una compra: calcula el total y descuenta el stock. Al separar estas acciones en archivos individuales, el código es fácil de mantener y escalar. Si mañana cambiamos la base de datos, las reglas de negocio no tendrían que modificarse.
5

Persistencia Local y Arquitectura de Datos

Room · Mappers · Inyección de Dependencias

Archivos Clave

  • data/local/ – Configuración de Room (SQLite)
  • data/mapper/ProductoMapper.kt – Traductor de datos
  • di/AppModule.kt – Inyección de Dependencias
  • DAOs y Entidades – Consultas SQL seguras

Conceptos Técnicos

  • Room – Base de datos local, Offline-first
  • Mappers – Transforman ProductoEntidad → Producto
  • Inyección de Dependencias – Instancias únicas, optimiza memoria
  • SQLite DAO Hilt
Guion (2 min)
Finalmente, explico la Persistencia Local y Arquitectura de Datos. Nuestra app es Offline-first gracias a Room, una base de datos local potente. Usamos DAOs para consultas SQL seguras. Un componente crítico son los Mappers: traducen los datos de la base al formato que entiende la lógica de negocio. Además, implementamos Inyección de Dependencias, permitiendo que toda la aplicación comparta una única instancia de la base de datos y los repositorios, optimizando la memoria y haciendo el sistema más estable.
